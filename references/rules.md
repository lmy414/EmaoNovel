# 规则引擎

**规则即法律。AI 执行任务前必须先读 `.rules/` 下的规则文件。**

---

## 规则架构

```
.rules/
├── 总规则.md           # 元规则：AI行为 + 加载顺序 + 冲突处理 + 禁止事项
│   ├── 引用 → 禁止词库.md
│   ├── 引用 → 敏感词库.md
│   └── 引用 → 自定义规则.md
├── 版本管理规则.md      # Git策略 + commit规范 + push时机
├── 结构规则.md          # 目录结构 + 输出格式 + 文风规范 + 人机分工
│   └── 引用 → 参考文风.md
├── 内容规则.md          # 叙事约束：角色关系、剧情红线、世界观一致性、术语
├── 自定义规则.md        # 用户扩展入口，可自行写规则或索引新文件
├── 禁止词库.md          # 绝对禁止的词汇
├── 敏感词库.md          # 需谨慎使用的词汇
└── 参考文风.md          # 文风参考文本，生成正文时按需加载
```

---

## 核心原则

1. **规则即法律**：每一条约束都是 AI 必须遵守的指令
2. **渐进加载**：总规则 → 核心规则（版本管理/结构/内容）→ 索引文件按需加载，避免上下文膨胀
3. **规则可随时修改**：修改后立即生效，纳入 Git 版本管理
4. **一项目一套规则**：换项目就换 `.rules/`，AI 行为立刻切换

---

## 设计理念

规则不是一次性填完的表格。项目初期只需 `总规则.md` + `版本管理规则.md` + `结构规则.md`（三个都由 `/nova init` 提供合理默认值）。`内容规则.md` 和 `自定义规则.md` 在写作过程中按需生长。

---

## 冲突处理

| 冲突类型 | 处理方式 |
|----------|----------|
| 规则 vs 规则 | 按优先级（总规则 > 版本管理 > 结构 > 内容 > 自定义） |
| 规则 vs 用户指令 | 高风险 → 拒绝并说明；低风险 → 提醒后执行 |
| 用户说"忽略规则" | 仅当次对话生效，下次对话恢复全量加载 |

## 自定义规则创建流程

**当用户说"加条规则"、"/nova rule"、"帮我定制规则"时，AI 执行以下流程：**

### 一、理解需求
1. 询问用户想约束什么
2. 引导描述具体场景
3. AI 提炼出可规则化的要点

### 二、推荐落点

| 需求类型 | 落点文件 | 示例 |
|----------|----------|------|
| 禁止特定词汇 | `禁止词库.md` | "禁用'卧槽'" |
| 统一写作风格 | `结构规则.md` 文风章节 | "短句为主，不超过 20 字" |
| 控制输出格式 | `结构规则.md` 输出格式章节 | "对话用破折号" |
| 角色关系红线 | `内容规则.md` 角色关系章节 | "男二不得与女主单独相处" |
| 剧情走向红线 | `内容规则.md` 剧情红线章节 | "不写悲剧结局" |
| 世界观铁律 | `内容规则.md` 世界观一致性章节 | "此世界没有复活术" |
| 术语统一 | `内容规则.md` 术语统一章节 | "灵力 → 不可写作法力" |
| 全新领域的规则 | `自定义规则.md` 索引新文件 | "新建战斗体系规则.md" |

### 三、生成与确认
1. AI 生成规则初稿
2. 展示给用户确认
3. 用户修改/补充
4. AI 写入对应文件
5. `git commit -m "规则：xxx"`

### 四、验证（可选）
1. 用新规则生成测试文本
2. 展示验证效果
3. 如需调整，返回第三步
