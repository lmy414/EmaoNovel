# 规则引擎

**规则即法律。AI 执行任务前必须先读 `.rules/` 下的规则文件。**

---

## 规则架构

```
.rules/
├── 总规则.md           # 元规则：AI行为 + 加载顺序 + 冲突处理 + 禁止事项
│   ├── 引用 → 禁止词库.md
│   ├── 引用 → 敏感词库.md
│   └── 引用 → 自定义规则.md
├── 版本管理规则.md      # Git策略 + commit规范 + push时机
├── 结构规则.md          # 目录结构 + 输出格式 + 文风规范 + 人机分工
│   └── 引用 → 参考文风.md
├── 内容规则.md          # 叙事约束：角色关系、剧情红线、世界观一致性、术语
├── 自定义规则.md        # 用户扩展入口，可自行写规则或索引新文件
├── 禁止词库.md          # 绝对禁止的词汇
├── 敏感词库.md          # 需谨慎使用的词汇
└── 参考文风.md          # 文风参考文本，生成正文时按需加载
```

---

## 核心原则

1. **规则即法律**：每一条约束都是 AI 必须遵守的指令
2. **渐进加载**：总规则 → 核心规则（版本管理/结构/内容）→ 索引文件按需加载，避免上下文膨胀
3. **规则可随时修改**：修改后立即生效，纳入 Git 版本管理
4. **一项目一套规则**：换项目就换 `.rules/`，AI 行为立刻切换

---

## 设计理念

规则不是一次性填完的表格。项目初期只需 `总规则.md` + `版本管理规则.md` + `结构规则.md`（三个都由 `/nova init` 提供合理默认值）。`内容规则.md` 和 `自定义规则.md` 在写作过程中按需生长。

---

## 冲突处理

| 冲突类型 | 处理方式 |
|----------|----------|
| 规则 vs 规则 | 按优先级（总规则 > 版本管理 > 结构 > 内容 > 自定义） |
| 规则 vs 用户指令 | 高风险 → 拒绝并说明；低风险 → 提醒后执行 |
| 用户说"忽略规则" | 仅当次对话生效，下次对话恢复全量加载 |

---

## 规则更新流程

**两种方式，效果相同：**

| 方式 | 示例 |
|------|------|
| 自然语言（推荐） | "增加规则：不写悲剧结局" |
| 命令 | `/nova rule` |

### 自然语言触发

用户说"增加规则：XXX"或"加条规则：XXX"，AI 自动完成以下三步——不需要用户指定放到哪里：

**第一步：识别内容 → 匹配落点**

| 你说... | AI 自动写入... |
|---------|---------------|
| "增加规则：禁用'卧槽'" | `禁止词库.md` → 自定义禁止词章节 |
| "增加规则：短句为主，不超过 20 字" | `结构规则.md` → 文风规范章节 |
| "增加规则：对话用破折号" | `结构规则.md` → 输出格式章节 |
| "增加规则：男二不得与女主单独相处" | `内容规则.md` → 角色关系约束章节 |
| "增加规则：不写悲剧结局" | `内容规则.md` → 剧情红线章节 |
| "增加规则：这个世界没有复活术" | `内容规则.md` → 世界观一致性章节 |
| "增加规则：灵力不能写成法力" | `内容规则.md` → 术语统一章节 |
| "增加规则：战斗升级体系" | `自定义规则.md` → 索引新文件 `战斗体系规则.md` |

**第二步：直接写入**

AI 将规则写入对应文件的对应章节，无需逐条确认。写入后简短告知用户写到哪个文件即可。

**第三步：commit**

`git commit -m "规则：xxx"`

### 修改已有规则

当用户说"把规则XXX改成YYY"或"删除规则XXX"，AI 全局搜索规则文件，找到匹配内容后执行修改或删除。

### 批量操作

当用户一次说多条规则时（如"增加规则：A、B、C"），AI 逐条识别 → 逐条写入对应文件 → 合并为一个 commit。

## Commit 规范

| 操作 | Commit 信息 |
|------|-------------|
| 新增禁止词 | `规则：新增禁止词汇` |
| 修订规则 | `规则：修订文风规范` |
| 新增自定义规则 | `规则：新增战斗体系规则` |
| 删除规则 | `规则：删除xxx规则` |
