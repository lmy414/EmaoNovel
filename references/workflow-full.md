# 工作流完整说明

详细版工作流文档，含设计理念与扩展方向。

## 概述

一套以 Git 为版本管理核心、AI 助手为执行层的小说创作工作流。全程在 OpenClaw 中通过对话完成——无需切换工具，无需手动操作文件系统，无需记忆 Git 命令。作者只需提供创意和决策，所有整理、归档、版本管理由 AI 自动化完成。

**运行环境**：OpenClaw 对话界面
**操作方式**：自然语言对话
**核心理念**：作者负责"写什么"，AI 负责"怎么存"。

---

## 实际工作流程

### 阶段一：原始素材投喂

**作者做什么：**
- 把零散的设定文件（txt、聊天记录、随手写的片段）放到项目目录下
- 或者直接口述设定，想到哪说到哪

**AI 做什么：**
- 读取所有素材文件，理解作品核心设定、角色关系、世界观
- 提取可结构化的信息，等待归档

**示例：**
作者丢进来三个 txt：一个写满角色设定碎片，一个写着剧情走向，一个记着世界观零散规则。AI 全部读完，整理出完整脉络。

---

### 阶段二：结构化归档

**AI 将原始素材整理为四份标准设定文件：**

| 文件 | 内容 |
|------|------|
| `作品名-作品定位.md` | 类型、尺度、感情线、CP、结局规划 |
| `作品名-世界观设定.md` | 世界观体系、政治/魔法/经济规则 |
| `作品名-角色设定.md` | 全部角色的详细档案（表格+分类） |
| `作品名-剧情大纲.md` | 分章/分阶段剧情大纲 |

**整理原则：**
- 保留作者原始素材文件不动
- 设定文件采用统一 Markdown 格式（表格 + 分级标题）
- 名称统一为 `作品名-分类.md`
- README.md 自动生成，含简介和目录

---

### 阶段三：Git 版本化

**初始化：**
```
git init
git add -A
git commit -m "作品名 世界书设定"
git branch -M main
```

**GitHub 私有仓库创建（自动化）：**
```
gh repo create 用户名/仓库名 --private --source=. --remote=origin --push
```

**Commit 规范：**

| 前缀 | 用途 | 示例 |
|------|------|------|
| `设定更新：` | 新增设定或情节 | `设定更新：新增反派角色线` |
| `设定修订：` | 修改已有设定 | `设定修订：世界观术语统一` |
| `正文：` | 新章节或续写 | `正文：第一章·初遇` |
| `修改：` | 微调 | `修改：主角性格描述` |
| `整理：` | 结构调整 | `整理：合并重复设定` |

---

### 阶段四：迭代修订

**作者只需口述修改意图，AI 自动完成：**

1. 全局搜索受影响位置
2. 列出需要修改的文件清单
3. 批量修改所有相关文件
4. `git commit` + `git push`

**典型场景：**
- "把公会系统改成佣兵体系" → AI 全局替换所有文件中的术语
- "主角的身世删掉那段" → AI 搜索并移除所有相关段落
- "这个角色性格我重新想了一下" → AI 更新角色设定文件

每次修改都是一条 commit，随时可回溯到任意版本。

---

### 阶段五：搁置与续写

- 暂停时：AI 在当前进度处标注，commit 并 push 到远端
- 续写时：AI 先读取全部设定文件和大纲，确认断点位置
- 状态永远不丢——远程仓库 + commit 历史 = 永久存档

## 目录结构约定

```
项目根目录/作品名/
├── README.md                  # 项目简介+目录
├── 作品名-作品定位.md
├── 作品名-世界观设定.md
├── 作品名-角色设定.md
├── 作品名-剧情大纲.md
├── .rules/                    # 项目级规则配置
│   ├── 文风规范.md             # 句式、节奏、用词偏好
│   ├── 输出格式.md             # 正文排版、对话格式、章节结构
│   ├── 角色命名规则.md          # 命名风格约束
│   └── 禁止事项.md             # 明确禁止的情节/写法
├── 原始素材/                   # 作者原始文件（保留不动）
├── 正文/                      # 正式章节
└── .git/
```

## 平台与工具

整个工作流在 **OpenClaw** 中通过自然语言对话完成。作者无需操作文件系统、无需记忆 Git 命令、无需手动配置 GitHub——所有技术细节由 AI 助手在对话中自动处理。

**各组件分工：**

- **作者**：负责创意输出。构思角色、设计世界观、规划剧情、做设定决策。只需要用自然语言表达想法，不碰任何技术层面。
- **AI 助手**：负责执行。读取素材、整理归档为结构化文档、操作 Git、管理 GitHub 仓库、全局搜索与批量修订。作者的每一句话，AI 将其转化为文件操作和版本记录。
- **OpenClaw**：AI 助手的运行平台。提供对话界面、文件读写能力、命令行执行能力。是作者和 AI 之间的桥梁——作者在聊天框里说，AI 在聊天框里做，所有操作在一个窗口内闭环。
- **Git**：版本管理核心。每次设定修改、每节正文推进，都生成一条带说明的 commit。任何时候可以回溯到任意历史版本，不怕改坏。
- **GitHub**：远程备份与协作基础设施。私有仓库存储所有文件，本地和远端双向同步。电脑坏了、换设备了、想拉人协作了——仓库一 clone 就能继续。
- **gh CLI**：GitHub 的命令行工具。AI 通过它自动创建私有仓库，无需作者打开网页手动操作。
- **SSH**：免密安全认证。本地和 GitHub 之间的加密通道，一次配置后 AI 可直接推送，无需每次输入密码。
- **Markdown (.md)**：设定文件存储格式。纯文本、结构清晰、Git 友好（可 diff 对比差异）、人机皆可读。相比 Word 或纯 txt，Markdown 在版本控制和格式化之间取得最佳平衡。

## 规则系统（可自定义）

每个项目可以通过 `.rules/` 目录下的规则文件自定义 AI 的行为。AI 在执行任何创作任务前会读取这些规则，确保输出风格、格式、内容边界完全符合作者预期。

**核心思路**：规则外置为文件，换项目就换规则。就像 AI 执行任务前读取技能文件（SKILL.md）一样——作者写的规则就是 AI 的"技能书"。

### 规则文件示例

| 规则文件 | 内容 | 效果 |
|------|------|------|
| `.rules/文风规范.md` | 句式偏好（短句/长句）、用词习惯、描写密度、对话节奏 | AI 生成的正文自动匹配作者文风 |
| `.rules/输出格式.md` | 章节标题格式、段落间距、对话标记（引号/破折号）、场景分割符 | 所有输出保持统一的排版结构 |
| `.rules/禁用词与禁忌.md` | 禁止使用的词汇、禁止描写的桥段类型、尺度边界 | AI 不会越界 |
| `.rules/命名规则.md` | 角色命名风格（中式/西式/奇幻）、地名前缀、专有名词表 | 世界观术语保持一致性 |
| `.rules/文件结构.md` | 自定义输出目录结构、文件命名格式（覆盖默认模板） | 不同项目类型用不同的组织方式 |
| `.rules/分镜模板.md` | AI 短剧分镜脚本的输出格式、镜头描述规范、提示词风格 | 直接对接 AI 视频生成流程 |

### 规则优先级

1. 项目级 `.rules/` 文件（最高优先级，仅影响当前项目）
2. 全局默认模板（AI 的内置规则，适用于未自定义的项目）
3. 作者实时口述指令（临时覆盖，不受文件限制）

### 使用方式

作者在项目初始化时写好 `.rules/` 下的规则文件，之后 AI 每一次操作都会自动遵循。想调整规则时，修改对应文件或口述"改一下文风规范"即可——AI 会更新规则文件并应用到后续所有生成中。

## 后续扩展

- **文风学习与模拟**：喂入参考文本，AI 学习作者句式、节奏、用词习惯
- **多格式输出**：游戏脚本格式、影视剧本格式、轻小说排版
- **角色关系图谱**：自动从设定中生成可视化角色关系网
- **时间线校验**：多线并行剧情自动检测时间矛盾
- **设定冲突检测**：修订时自动提示与已有内容的冲突
- **一键导出**：按出版/投稿需求导出 PDF、EPUB、Word 等格式
- **分镜脚本生成**：从剧情大纲直接生成 AI 视频描述词（适用于 AI 短剧制作）
- **多渠道适配**：同一设定库输出不同尺度/平台的版本
