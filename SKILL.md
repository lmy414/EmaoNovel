以 Git 为版本管理核心、AI 为执行层。作者负责"写什么"，AI 负责"怎么存"。

作者说什么 | AI 做什么
---|---
"帮我整理这些设定" / "这是角色设定" / 丢素材文件 | 进入阶段一→二：读取素材→整理为结构化文档
"初始化小说项目" / "创建仓库" | 进入阶段三：git init → 创建 GitHub 私有仓库 → 推送
"把XX改成YY" / "这个设定重想" / "删掉那部分" | 进入阶段四：全局搜索→批量修改→commit+push

"改下文风规范" / "加条规则" | 修改 `.rules/` 对应文件→commit+push

## 五阶段流程

### 阶段一：原始素材投喂

作者丢素材（文件或口述），AI 全量读取并理解。

**动作：**
1. 读所有素材文件
2. 提取可结构化信息：角色、世界观、剧情走向、核心设定
3. 准备归档

---

### 阶段二：结构化归档

整理为四份标准 Markdown 文件，存于 `作品名/` 目录。

| 文件 | 内容 |
|------|------|
| `作品名-作品定位.md` | 类型、尺度、感情线、CP、结局规划 |
| `作品名-世界观设定.md` | 世界观体系、规则、背景 |
| `作品名-角色设定.md` | 全部角色详细档案（表格+分类） |
| `作品名-剧情大纲.md` | 分章/分阶段剧情大纲 |

**规则：**
- 原始素材保留不动 → `作品名/原始素材/`
- 设定文件统一 Markdown（表格 + 分级标题）
- 名称格式：`作品名-分类.md`
- 自动生成 README.md（简介+目录）

完成后进入阶段三。

---

### 阶段三：Git 版本化

```
cd 作品名
git init
git add -A
git commit -m "作品名 世界书设定"
git branch -M main
gh repo create 用户名/仓库名 --private --source=. --remote=origin --push
```

**Commit 规范：**

| 前缀 | 用途 | 示例 |
|------|------|------|
| `设定更新：` | 新增设定或情节 | `设定更新：新增反派角色线` |
| `设定修订：` | 修改已有设定 | `设定修订：世界观术语统一` |
| `正文：` | 新章节或续写 | `正文：第一章·初遇` |
| `修改：` | 微调 | `修改：主角性格描述` |
| `整理：` | 结构调整 | `整理：合并重复设定` |

---

### 阶段四：迭代修订

作者口述修改意图 → 全局搜索 → 批量修改 → commit + push。

**流程：**
1. 理解修改意图
2. 搜索所有受影响文件，列出修改清单
3. 逐个修改
4. `git add -A` + `git commit -m "前缀：说明"` + `git push`

每次修改一条 commit，随时可回溯。

---

### 阶段五：搁置与续写

**搁置：** 标注当前进度，commit + push 到远端。

**续写：**
1. `git pull` 拉最新
2. 读全部设定文件 + 大纲
3. 确认断点位置
4. 从断点继续

## 目录结构

```
项目根目录/作品名/
├── README.md                    # 项目简介+目录
├── 作品名-作品定位.md
├── 作品名-世界观设定.md
├── 作品名-角色设定.md
├── 作品名-剧情大纲.md
├── .rules/                      # 项目级规则（可选）
│   ├── 文风规范.md
│   ├── 输出格式.md
│   ├── 禁用词与禁忌.md
│   └── 命名规则.md
├── 原始素材/                     # 作者原始文件（保留不动）
├── 正文/                        # 正式章节
└── .git/
```

## .rules/ 规则系统

项目 `.rules/` 存放自定义规则。AI 在每个创作任务前读取。

**优先级：** 项目 `.rules/` > 全局默认 > 作者实时口述

模板见 `references/rules-templates.md`。
